# Requirements Authoring & Management (RAM) — Docs Repo

This repository contains the requirements documents for the **Requirements Authoring & Management (RAM) Tool** — a graph-first, model-driven requirements IDE for software engineering and senior design courses at TCU.

The five numbered Markdown files in this repo **are** the deliverables. There is no separate source code, build system, or test suite. They are the single source of truth for the project's requirements.

## Quick start

1. Clone the repo.
2. Install [Claude Code](https://claude.com/claude-code) (optional but strongly recommended — see below).
3. Open the repo in your editor of choice. Edit the relevant `.md` file.
4. Run `/build` in Claude Code to verify and resync numbering, TOCs, and cross-doc consistency.
5. Review the report, fix any flagged semantic issues, and commit.

If you don't have Claude Code, see the [Without Claude Code](#without-claude-code) section.

## The five documents

The files are intentionally ordered and form a coherent set. Read and edit them in order:

1. **`project-glossary.md`** — domain vocabulary; canonical definitions.
2. **`vision-and-scope.md`** — business objectives (BO-*), risks (RI-*), assumptions (AS-*), major features.
3. **`use-cases.md`** — behavioral specifications ("The Student …", "The Instructor …").
4. **`business-rules.md`** — cross-cutting policies, constraints, and access rules (BR-*) cited by the use cases and the SRS.
5. **`software-requirements-specification.md`** — architecture, functional requirements (FR-*), quality attributes.

These describe the same underlying system from different angles. **A change in one usually requires updates in others** (especially glossary terms used across multiple docs). See [Cross-document coherence](#cross-document-coherence) below.

## Authoring conventions

To keep the docs consistent and exportable, all five files follow the same conventions.

### Heading numbering

- H1 → `# **N. Title**` — e.g., `# **1. Introduction**`
- H2 → `## **N.M Title**` — e.g., `## **1.1 Background**`
- H3 → `### *N.M.K Title*` — e.g., `### *4.2.1 Graph-First Requirements Model*`

The number goes **inside** the bold/italic markers. The doc title and the "Table of Contents" heading are **not** numbered.

**Two exceptions to the numbering scheme:**

- **`use-cases.md` is not section-numbered.** It's a catalog grouped by area: unnumbered H2 area headings (`## **Glossary**`, `## **AI Assistants**`, …) with each use case an H3 carrying a stable area-prefixed ID — `### **UC-GLO-1: The Student views…**`. The ID format is `UC-<AREA>-<n>` (append-only within each area); these IDs are stable handles and are **never renumbered**.
- **Glossary *term* headings are unnumbered.** In `project-glossary.md` the Section 1/2 headings keep the `N.M` scheme, but the individual term headings under Definitions are a flat, unnumbered catalog — `## **Template**`, not `## **2.13 Template**` — so terms can be inserted or reordered without renumbering. Reference terms by name, never by section number.

### Anchors

Heading anchors are **auto-generated by the renderer** (Typora, GitHub) from the heading text — headings do **not** carry explicit `{#slug}` attributes. The kramdown `{#id}` syntax isn't honored by Typora and shows up as clutter in its Outline panel, so we don't use it.

The slug the renderer computes from a heading is: lowercase the rendered text (including the number prefix), replace whitespace with `-`, strip everything that isn't alphanumeric or a hyphen, and do **not** collapse consecutive hyphens. Examples:

| Heading | Slug |
| --- | --- |
| `## **1.1 Background**` | `#11-background` |
| `## **Graph-First Model**` (unnumbered glossary term) | `#graph-first-model` |
| `### *4.2.1 Graph-First Requirements Model*` | `#421-graph-first-requirements-model` |
| `## **4.2 Major Features / Scope**` | `#42-major-features--scope` (consecutive hyphens not collapsed) |

**Avoid Markdown-significant characters in heading text** (`[`, `]`, backticks, `<>`, `!`). Typora interprets `[...]` inside a heading as reference-link syntax and computes a non-obvious slug, even when the brackets are escaped. If you need brackets in a section name, rewrite the heading without them.

Slugs change when sections are renumbered — `/build` regenerates the TOC each run, so this stays in sync automatically.

### Table of Contents

Each doc has a TOC at the top, formatted as a nested Markdown list (two spaces of indent per level, all heading levels included, no page numbers):

```markdown
# **Table of Contents**

- [1. Introduction](#1-introduction)
  - [1.1 Background](#11-background)
  - [1.2 Current Process Flows](#12-current-process-flows)
- [2. Business Requirements](#2-business-requirements)
  - [2.1 Business Opportunity/Problem Statement](#21-business-opportunityproblem-statement)
  ...
```

### Functional Requirement IDs

A **use case is itself a high-level functional requirement** (SRS §5.1) — its "System shall" steps + Associated Information are its detailed spec, so use-case CRUD flows are *not* restated as FRs. SRS **§5.2** holds only the non-use-case, system-level behaviors, with IDs in the format `FR-<AREA>-<n>` (descriptive area code, hyphen, number), **parallel to the `UC-<AREA>-<n>` use-case IDs** and **independent of section numbering**:

| Prefix | Category |
| --- | --- |
| `FR-SAVE-*`  | Autosave |
| `FR-LOCK-*`  | Locking |
| `FR-COL-*`  | Collaboration |
| `FR-VAL-*`  | Validation (ReqLint) |
| `FR-AI-*` | AI |
| `FR-TPL-*`  | Templates |
| `FR-GLO-*`  | Glossary (terminology invariants) |
| `FR-HIS-*`  | History & authorship metadata |
| `FR-SEC-*`  | Security |
| `FR-EXP-*`  | Export |
| `FR-IMP-*`  | Import |
| `FR-PERF-*`  | Performance |
| `FR-NOT-*`  | Notifications |

When you add a §5.2 FR, reuse the matching area — introduce a new descriptive area code only for a genuinely new non-use-case category. FR IDs are append-only within their area and do not change when sections are renumbered.

### Business Rule IDs

Business rules live in `business-rules.md` as a single append-only sequence (`BR-1`, `BR-2`, …), grouped thematically into numbered §2 subsections. Like FR IDs, `BR-*` is an identifier space **independent of section numbering** — never renumber a rule when sections move. Use cases cite rules by ID in their **Business Rules** field, and the SRS cites them where an FR or quality attribute enforces one.

### Writing style (ReqLint)

Functional requirements should be atomic, testable, and unambiguous. Avoid:

- **Vague verbs:** `manage`, `support`, `handle`, `process`, `deal with`
- **Subjective adjectives:** `fast`, `user-friendly`, `intuitive`, `easy`, `quick`, `appropriate`, `seamless`
- **Missing "shall":** FR statements should be of the form "The system shall …"
- **Compound requirements:** one FR, one behavior

### Glossary term formatting

Glossary terms are not visually marked every time they appear. In normal prose, write defined terms as plain lowercase text, not bolded or Title Case: `student`, `instructor`, `course admin`, `course section`, `document section`, `requirement artifact`, `artifact link`, `business objective`, `functional requirement`.

Preserve capitalization for headings and use-case titles, sentence starts, acronyms and proper nouns (`RAM`, `Project Pulse`, `AI`, `FERPA`, `EARS`, `INVEST`, `WCAG`, `Wiegers`, `Socratic`), document titles (`Vision and Scope`, `Project Glossary`, `Use Cases`, `Business Rules`, `Software Requirements Specification`), UI labels, table column headings, and code identifiers or enum values (`RequirementArtifactType`, `FUNCTIONAL_REQUIREMENT`, `DERIVES_FROM`). If a term names a specific document or named document view, use the document title form (`Use Cases document`, `Business Rules document`); if it names the concept in ordinary prose, use lowercase (`a use case`, `the business rules`). Artifact-type vocabulary uses two registers: lowercase in normal prose, but Title Case when it is functioning as a controlled label in taxonomy lists, artifact-type tables, enum mapping prose, or template/placement matrices.

## Keeping the docs in sync — the `/build` command

This repo treats the documents as artifacts that need to "compile": numbering sequential, TOCs in sync with body, anchors resolving, terminology coherent across files.

The `/build` slash command (defined in [`.claude/commands/build.md`](.claude/commands/build.md)) automates this. In Claude Code, type `/build` and it will:

1. **Auto-fix mechanical issues** in place:
   - Renumber headings if any are out of sequence.
   - Strip any leftover `{#slug}` attributes from heading lines (kramdown anchors aren't honored by Typora — see [Anchors](#anchors) above).
   - Regenerate each doc's TOC to match the current body, using slugs that match what Typora auto-generates.
   - Verify TOC anchors resolve.
2. **Report semantic issues** for your review (no auto-edits):
   - Glossary terms used in docs 2–5 but not defined in doc 1.
   - References to FR IDs that don't exist.
   - ReqLint-style writing issues in FRs.
   - Cross-doc terminology drift (the same concept named differently in different docs).

**Recommended workflow:**

```
edit a doc manually  →  run /build  →  review report  →  fix flagged issues  →  commit
```

## Without Claude Code

If you don't have Claude Code installed, you'll need to do the maintenance manually. The key things to remember:

1. After adding/removing/renaming a heading, walk down the doc and **renumber sequentially** following the scheme above. (Exceptions: never renumber `use-cases.md` — its `UC-<AREA>-<n>` IDs are stable handles — and don't number the glossary's term headings.)
2. **Regenerate the TOC** at the top of the file so it matches the new structure. Each TOC entry's slug must match what Typora auto-generates from the heading text (see the slug table above).
3. If you renamed or renumbered a heading, **its slug changed** — update the matching TOC entry. Also grep the other four docs for references to the old text — cross-doc references need updating too.

This is tedious by hand. If you'll be editing the docs regularly, installing Claude Code and using `/build` is much faster and less error-prone.

## Cross-document coherence

The five docs describe one system from different angles. When you change one, **always grep the others**:

- Renaming a **glossary term**? It probably appears in 2, 3, 4, and 5. Update all references.
- Renaming a **feature**? It appears in Vision and Scope, has related use cases in doc 3, and FRs in doc 5.
- Changing an **FR ID**? Search prose for `FR-EXP-1`-style references.
- Changing a **BR ID**? It's cited in use-case **Business Rules** fields and the SRS.
- Renaming an **actor** (e.g., "Student" → "Author")? Sweep all five docs.

The `/build` command surfaces obvious cross-doc drift, but a quick grep before committing is good hygiene either way.

## Notes and known quirks

- **Revision History tables** at the top of each doc are placeholder rows. Fill them in when you finalize a version.
- **Square-bracketed italic text** (e.g., `*[Note: ...]*`, `*[Describe ...]*`) is author-guidance from the Wiegers/Beatty template. Leave it in place until you finalize the doc.
- **Google Docs export residue.** Older/imported text may still surface artifacts from the original Google Docs source, such as anchor syntax (`{#heading=h.y0hfekdupgbe}`), image placeholders (`![][image1]`), or raw URL stubs. If you encounter them while editing nearby content, clean them up opportunistically.
- **`software-requirements-specification.md` Section 4** (System Context / Container Diagram) describes *Project Pulse* — the Vue.js + Spring Boot + relational DB + Gmail platform — as the host environment. This is correct, not stale: RAM is a module inside Project Pulse and relies on its existing course/section/team/auth infrastructure. Section 4 presents the RAM module's views, endpoints, and tables as extensions of that shared architecture, plus the external LLM service it integrates with.
