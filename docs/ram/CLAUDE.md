# CLAUDE.md

This file governs edits to the **`docs/ram/` spec subtree** — the requirements documentation for the Requirements Authoring & Management (RAM) module. It applies to anything under `docs/ram/`.

## What this directory is

`docs/ram/` is the **spec→design→trace chain for the RAM module** — the requirements documentation that drives its implementation. *This* file governs only **how those docs are authored and kept consistent**: numbering, TOCs, anchors, ID schemes, file relationships, and cross-doc coherence. The repo-root `CLAUDE.md` governs the codebase as a whole and explains how this spec drives the code (its **Spec-driven RAM development** section).

Layout of `docs/ram/`:

- `requirements/` — the five core spec docs (below).
- `design/` — design docs generated from the spec, one per UC area; they sit *below* the SRS and cite the UC/FRs they realize without restating them.
- `traceability.md` — the spec→code matrix: one row per use case → FR IDs → design doc → frontend/backend modules → tests → status.
- `guides/` — supporting authoring/build guidance that isn't itself a spec doc.

The five core Markdown files under `requirements/` form a coherent set, best read in this order:

1. `requirements/project-glossary.md` — domain vocabulary; canonical definitions
2. `requirements/vision-and-scope.md` — business objectives (BO-*), risks (RI-*), assumptions (AS-*), major features
3. `requirements/use-cases.md` — behavioral specs ("The Student …", "The Instructor …")
4. `requirements/business-rules.md` — cross-cutting policies, constraints, and access rules (BR-*) cited by the use cases and the SRS
5. `requirements/software-requirements-specification.md` — architecture, functional requirements (FR-*), quality attributes

Below, the five core docs are referred to by bare filename.

## Conceptual model to internalize before editing

The five spec docs are independent Markdown files, but they all describe **one shared model** of the RAM domain — the same artifacts, terms, features, and rules seen from different angles. (The model itself is defined in the spec — start with the core graph model terms in `project-glossary.md` — not re-explained here.) So when changing one doc, check whether the same concept appears in another — they're supposed to agree because they describe the same underlying model:

- A **glossary term** added/renamed in `project-glossary.md` should be used consistently across docs 2–5. Don't introduce a synonym for an already-defined term.
- A **feature** mentioned in `vision-and-scope.md` typically maps to use cases in `use-cases.md` and functional requirements in `software-requirements-specification.md`.
- **A use case is itself a (high-level) functional requirement.** That is why `use-cases.md` is referenced from SRS §5.1 *Use Cases*, under Functional Requirements. A use case's "The system …" steps plus its **Associated Information** are its detailed functional spec — so **do not restate a use case's flow (CRUD: create/view/edit/delete/search) as a separate FR.** When a use-case step needs more detail, elaborate it in that use case's Associated Information, not in a new FR. SRS **§5.2 is the complement**: only non-use-case, system-driven, event-driven, global, or background behaviors (autosave, locking, validation engine, real-time collaboration, AI/LLM orchestration, templates, terminology invariants, versioning, authorship metadata, security, export formatting, import, performance). The litmus test before adding a §5.2 FR: *is there a use case whose system-subject step already says this?* If yes, it belongs in that use case (or its Associated Information), not §5.2 — keep the bright line so §5.2 never re-describes use-case behavior. Because the **use case is the unit of citation** — `traceability.md` carries one row per UC and tests are tagged to the UC, with no finer per-step handle — keep each use case right-sized so "UC-X is done / passing" is a meaningful statement (split a use case that grows large enough to half-satisfy rather than giving its steps sub-IDs), and write **extensions** precisely: with no step-level IDs, a use case's extensions plus their Associated Information are the only record of its negative/exception behavior, so they carry the edge-case requirements.
- §5.2 functional requirement IDs use the format `FR-<AREA>-<n>` — a descriptive area code, a hyphen, then a number — deliberately **parallel to the `UC-<AREA>-<n>` use-case IDs**, and an identifier space independent of section numbering. The EARS-style §5.2 areas are: `FR-SAVE-*` autosave, `FR-LOCK-*` locking, `FR-COL-*` collaboration, `FR-VAL-*` validation, `FR-AI-*` AI, `FR-TPL-*` templates, `FR-GLO-*` glossary (terminology invariants), `FR-HIS-*` history & authorship metadata, `FR-SEC-*` security, `FR-EXP-*` export, `FR-IMP-*` import, `FR-PERF-*` performance, `FR-NOT-*` notifications. Reuse an existing area when adding a §5.2 FR; introduce a new descriptive area code only for a genuinely new non-use-case category, and never renumber an existing FR.
- **Business rule IDs (`BR-*`)** live in `business-rules.md` and are a single append-only sequence (`BR-1`, `BR-2`, …), grouped thematically into numbered §2 subsections. Like FR IDs, they are an **identifier space independent of section numbering** — never renumber a `BR-*` when sections move. Use cases cite them by ID in their **Business Rules** field; the SRS cites them where an FR or quality attribute enforces one.

## Authoring conventions

These files are the single source of truth for the project's requirements. Numbering, TOCs, anchors, and cross-doc terminology must stay consistent as the docs evolve.

**Numbering scheme** for the section-numbered docs — `project-glossary.md`, `vision-and-scope.md`, `business-rules.md`, `software-requirements-specification.md` (doc title and "Table of Contents" are NOT numbered):
- H1 → `# **N. Title**`
- H2 → `## **N.M Title**`
- H3 → `### *N.M.K Title*`

**Exception — `use-cases.md` does NOT use the section-numbering scheme above.** It is a catalog of use cases **grouped by area**, not a hierarchical spec, and its IDs deliberately **decouple identity from document position** (a single global sequence was abandoned because inserting one use case renumbered everything after it and broke every cross-reference). Structure: the top-level heading (Use Case List) stays unnumbered H1; under **Use Case List**, use cases are grouped under **unnumbered H2 area sub-headings** (`## **Glossary**`, `## **AI Assistants**`, …), and each use case is an **H3** heading carrying an **area-prefixed ID** — `### **UC-GLO-1: The Student views…**`. The ID format is `UC-<AREA>-<n>`, where `<AREA>` is a stable short area code (3 letters for most areas, 2 for `AI`) and `<n>` is assigned **append-only within that area** (no global numbering, no document-order contiguity requirement). Areas, in document order: `TPL` templates/provisioning, `GLO` glossary, `DOC` requirement documents, `ART` requirement artifacts, `LNK` artifact links & tracing, `VAL` validation, `COL` collaboration, `REV` review & submission, `EXP` export, `CFG` AI configuration, `AI` AI assistants. The same `UC-<AREA>-<n>` appears in the `UC ID and Name:` table field, and inline `UC-<AREA>-<n>: <name>` cross-references must agree. **When adding a use case, append it to its area** — take the next free `<n>` in that area, drop the H3 block under the matching area sub-heading, never insert mid-sequence or renumber existing IDs (the whole point is that IDs are stable handles). Need a brand-new area? Pick an unused short (preferably 3-letter) code and place it sensibly. Then regenerate this doc's grouped TOC (Use Case List → area → UC). `/build` will NOT impose the `# **N. Title**` scheme here and will NOT renumber UC IDs; it strips stray anchors, regenerates the grouped TOC, reconciles TOC↔body, and flags **per-area** sequencing problems (gaps or duplicates *within* an area) and any ID↔`UC ID and Name:` mismatch. Never let any tool collapse these IDs into a global sequence.

**Anchors are auto-generated by the renderer** (Typora, GitHub). Headings should NOT carry explicit `{#slug}` attributes — kramdown `{#id}` syntax is not honored by Typora and shows up as clutter in its Outline panel. Strip any leftover `{#xxx}` tokens. The slug the renderer computes from a heading is: lowercase the rendered text (including the number prefix when the heading carries one), replace whitespace with `-`, strip everything that isn't `[a-z0-9-]`, do NOT collapse consecutive hyphens. Examples: `## **1.1 Background**` → `#11-background`; `## **Graph-First Model**` (unnumbered glossary term) → `#graph-first-model`; `## **4.2 Major Features / Scope**` → `#42-major-features--scope`. Slugs change when sections are renumbered — `/build` regenerates the TOC every run, so they stay in sync automatically.

**Square brackets and other Markdown-significant characters in heading text are a known trap.** Typora interprets `[...]` inside a heading as reference-link syntax and computes a non-obvious slug, even when escaped. Prefer rewriting such headings (e.g., drop the brackets) over trying to derive the matching slug. Same caution for backticks, `<>` HTML, and `!`.

**TOC format** (consistent across all five docs): `# **Table of Contents**` heading, then a nested Markdown bullet list (two-space indent per level), all heading levels included, entries as `- [N.M Title](#slug)` where slug follows the auto-gen rule above.

**List style.** Use `-` (hyphen) for every unordered-list marker, in both the TOC and body content, at all nesting levels — not `*` or `+`.  Only convert leading list markers — never touch `*emphasis*`, `**bold**`, or `\*\*\*` placeholders.

**Glossary terms are unnumbered; group them thematically.** In `project-glossary.md` the section headings keep the standard scheme (`# **1. Introduction**`, `## **1.1 Purpose**`, `# **2. Definitions**`), but the individual term headings under Definitions are **unnumbered** — `## **Template**`, not `## **2.13 Template**`. The term list is therefore a flat catalog whose identity is the term name, not a position (the same identity-decoupled-from-position principle as the area-prefixed IDs in `use-cases.md`): terms can be inserted or reordered freely with **no renumbering**. Related concepts must still sit next to each other — when adding a term, drop it into its thematic group, don't just append at the end. Current groups (in order): core graph model (artifacts, types, keys, links, traceability, navigation); documents (document, document section, section key, template, authoring destination); requirement types (objective, risk, rule, feature, use case, FR, constraint, quality attribute, external interface, glossary term, stakeholder, assumption, dependency); authoring & validation (ReqLint, keyword); AI & teaching (assistant, assistant instructions, teaching context, project source material); users & organization (team, course, course section, course admin, instructor, student); acronyms & standards (MVP, FERPA, EARS, INVEST, Wiegers Requirement Categories, SSO, LLM, WCAG). Reference glossary terms by name, never by section number (e.g., "distinct from document section", not "see 2.11") — names are stable, numbers are positional and rot on every insertion. A numbered-anchor link like `[Document Section](#document-section)` works because the slug now derives from the name, not a number. Section keys (`vision.businessObjectives` etc.) are stable identifiers, unaffected by any of this.

**Glossary terms in prose use plain lowercase.** Do not bold or Title-Case glossary terms merely because they are defined terms. In ordinary prose, write `student`, `instructor`, `course admin`, `course section`, `document section`, `requirement artifact`, `artifact link`, `business objective`, `functional requirement`, and similar terms as normal lowercase text. Preserve capitalization for headings and use-case titles, sentence starts, acronyms and proper nouns (`RAM`, `Project Pulse`, `AI`, `FERPA`, `EARS`, `INVEST`, `WCAG`, `Wiegers`, `Socratic`), document titles (`Vision and Scope`, `Project Glossary`, `Use Cases`, `Business Rules`, `Software Requirements Specification`), UI labels, table column headings, and code identifiers or enum values (`RequirementArtifactType`, `FUNCTIONAL_REQUIREMENT`, `DERIVES_FROM`). If a term names a specific document or named document view, use the document title form (`Use Cases document`, `Business Rules document`); if it names the concept in ordinary prose, use lowercase (`a use case`, `the business rules`). Artifact-type vocabulary uses two registers: lowercase in normal prose, but Title Case when functioning as a controlled label in taxonomy lists, artifact-type tables, enum mapping prose, or template/placement matrices.

**"Section" is ambiguous — never write it bare.** Two domain senses collide: course section (a Course offering that students enroll in and that groups them into teams — the Project Pulse org unit; assistants and teaching context are configured per course section) and document section (a template-defined part of a requirement document — the unit that gets locked, validated, and versioned). Always qualify a domain mention as one or the other; never leave a bare "Section"/"section" for a domain concept. The *only* unqualified "section" allowed is a document referring to its own numbered parts ("see Section 4.2", "this section captures…") and the Wiegers template's bracketed author-guidance. Note: `Section Key` is a defined term (it maps artifacts to document sections) and stays as-is; `section-based requirement document` keeps its compound adjective. Rule of thumb when classifying: course section = enrollment / instructors / students / assistant enablement / teaching context; document section = document structure / locking / validation / authoring. Related but on a different axis: authoring destination is the glossary umbrella term for "a document section or a use case" — the lockable unit an authored edit or an accepted AI candidate is applied to (e.g., the structuring assistant in UC-AI-4 targets one). Use it (short form "destination") instead of coining a synonym; note a use case is a destination but is *not* a document section, so don't fold the two together.

**Cross-doc propagation.** Glossary terms, FR IDs, BR IDs, feature names, and actor names span multiple files. When you rename or restructure something in one doc, grep the *whole repo* (not just the other four core docs) before finalizing — see the dependency map below.

**Cross-document dependency map.** Requirements here are spread across files that reference each other, so a change in one place usually obligates a check elsewhere. When you change the left column, re-sync the right. Don't trust memory — run `/build`, which mechanically verifies the checkable couplings (UC↔traceability rows, FR-ID resolution, BR-ID resolution). Note that `traceability.md` is a **sixth** coupled file beyond the five core docs; it is the most-forgotten dependency precisely because it lives outside the `requirements/` set.

| When you change… | …check / re-sync |
|---|---|
| a `UC-<AREA>-<n>` in `use-cases.md` (add to an area / rename / remove) | `traceability.md` (exactly one row per UC, in UC-area order), the SRS (every `FR-*` the UC cites must exist), `project-glossary.md` (terms used), `vision-and-scope.md` (the feature/actor it realizes), and regenerate the grouped use-cases TOC |
| an `FR-*` in `software-requirements-specification.md` (add / renumber / remove) | use cases that cite it (`Realized by` / `honors` lines), `traceability.md` (`FR IDs` column), `vision-and-scope.md` features, glossary terms it leans on |
| a `BR-*` in `business-rules.md` (add / reword / remove) | use cases that cite it in their **Business Rules** field, the SRS where an `FR-*` or quality attribute enforces it, glossary terms it leans on |
| a glossary term in `project-glossary.md` (add / rename / delete) | every use of it across docs 2–5 (no synonyms, no dangling references), and the glossary TOC |
| a feature / `BO-*` / `RI-*` / `AS-*` in `vision-and-scope.md` | the use cases and `FR-*`s that implement it; the glossary if it introduces a term |

The two couplings `/build` checks deterministically are **`UC-<AREA>-<n>` ↔ traceability row** (Phase 2 step 7) and **`FR-*` reference resolution** (Phase 2 step 2, repo-wide including `traceability.md`). The rest are judgment calls `/build` surfaces but won't auto-fix.

**Diagrams.** Author diagrams as Mermaid fenced code blocks (` ```mermaid `). Mermaid is plain text — diffable, version-controllable, editable directly, and renders natively in Typora and GitHub. Keep diagram content consistent with the surrounding prose. (Mermaid C4 notation — `C4Context` / `C4Container` — is nicer for architecture views but is experimental and depends on the Mermaid version Typora bundles; confirm it renders before relying on it, and keep a plain `flowchart` fallback in mind.)

**The `/build` slash command** (in `.claude/commands/build.md`) sweeps all five docs: renumbers, strips leftover `{#slug}` attributes, regenerates TOCs with auto-gen slugs, and reports semantic issues (undefined glossary terms, dangling FR/BR refs, ReqLint violations, cross-doc drift). Suggest it to the user after any non-trivial structural edit; run it when they ask for a "build" / "verify" / "resync" / "check" pass. **`use-cases.md` and `project-glossary.md` are handled specially** (see the numbering exception and the glossary-terms convention above): `/build` never renumbers their area-prefixed UC IDs or unnumbered term headings, but still strips anchors, regenerates their TOCs, reconciles TOC↔body, and runs the semantic + per-area sequencing checks.

## Known quirks

- **Don't treat the Project Pulse references in the SRS as stale.** SRS Section 4 (System Context / Container Diagram) and Operating Environment describe Project Pulse as RAM's host environment — that is correct, not stale (RAM is a module inside it; see [`README.md`](README.md)). Don't refactor those references away or duplicate the architecture "for RAM"; author any RAM-specific addition (new SPA views, API endpoints, DB tables for the requirements graph, the LLM service as an external integration) as an extension of the existing description.
- **Google Docs export residue.** Some older/imported text may still surface image placeholders like `![][image1]`, `[URL](...)` link stubs, or `{#heading=h.xxxx}` anchors. If you encounter them while editing nearby content, clean them up opportunistically; explicit heading anchors should be stripped — see Authoring conventions (no explicit anchors on headings).
- **Revision History tables** at the top of each doc are placeholders (`<dd/mmm/yy>`, `<x.x>`). Only fill them in when the user explicitly asks.
- **Square-bracketed italic blocks** (e.g. `*[Note: ...]*`, `*[Describe ...]*`) are author-guidance from the Wiegers/Beatty template. Leave them in place when editing surrounding content unless the user is finalizing the doc.

## Working in `docs/ram/`

- No traditional build, lint, or test commands for the spec docs — but the `/build` slash command (see Authoring conventions above) acts as their build-and-verify pass. "Correctness" means cross-document consistency and adherence to the templates the docs themselves describe (ReqLint-style: no vague verbs like "manage"/"support", no subjective adjectives like "fast"/"user-friendly", "shall" statements for FRs, atomic and testable).
- The Use Cases doc is large (~88 KB). Prefer targeted reads (offset + limit, or Grep) over reading it whole.
- The `.claude/` folder at the repo root holds project-local settings and the `/build` and `/feature` slash commands — leave it alone unless the user asks to change permissions, hooks, or commands.

## Other files under `docs/ram/` (beyond the five core docs)

- **`traceability.md`** — the live UC ↔ FR ↔ code/test matrix, the spec→code map. **Coupled to the core docs** (the sixth coupled file in the dependency map — one row per UC, `FR IDs` column); keep it in sync. `/feature` updates the relevant row in its Phase 5; `/build` verifies it.
- **`requirements/OPEN-ISSUES.md`** — the working backlog of everything still needed to make the docs implementation-ready. Items carry stable append-only `OI-n` IDs grouped by priority (P0–P3); check the box and add a one-line note when one is done, append the next free `OI-n` for new gaps. When the user says "let's do OI-3" or asks what's left, this is the source; check the file itself for current status rather than relying on examples in guidance docs. Owner tags: ✍️ Claude can draft, 🧑 needs a user decision, 🔢 needs a value only the user has.
- **`README.md`** — a short human-facing pointer into `docs/ram/` (layout + where the conventions live). It deliberately does **not** restate the authoring conventions — those live here, in this file — so it isn't a drift risk; keep it that way.
- **`design/`** — design docs generated by `/feature`, one per UC area, sitting below the SRS (component/class design, sequence diagrams, API contracts, DB schema). `design/README.md` holds the conventions (one doc per area, cite the UC/FRs, never restate them); the per-area docs are added as each area is implemented.
- **`guides/`** — supporting authoring/build guidance, not spec docs:
  - `ai-implementation-notes.md` — build guidance for the AI Assistants.
  - `cross-document-review-criteria.md` — criteria + system prompt for the whole-project critique (see `OI-16`).
